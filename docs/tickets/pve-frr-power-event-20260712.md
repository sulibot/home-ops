# Incident: site power event + SDN config regeneration took down the fabric (2026-07-12)

## Impact

~20:15 site power event rebooted all three PVE nodes simultaneously. On boot
the fabric did not re-form: no corosync quorum, all Ceph daemons crash-looped,
HA frozen, cluster-101 fully down (all 6 VMs stopped, then unstartable).
cluster-104 (physical, non-PVE) and VLAN-attached LXCs (minio, zot, nixtest01)
recovered on their own. Total cluster-101 outage ≈ 1h15m; recovery was manual.

## Root causes (two, compounding)

1. **PVE SDN regenerates `/etc/frr/frr.conf` at boot**, keeping only the
   EVPN overlay it manages. The hand-built underlay (OSPF over the 25G mesh,
   iBGP over loopbacks, EDGE eBGP to RouterOS) survives only if present in
   `/etc/frr/frr.conf.local`, which PVE merges in.
   - pve01 had **no** `frr.conf.local` at all.
   - pve02/pve03 had **stale Dec-2025** ones (old AS 65001 design, obsolete
     addressing) — merged, but useless.
   The live config had been running in-memory/hand-written; the reboot erased it.
2. **PVE also regenerates `/etc/frr/daemons` at boot with `ospfd=no`.**
   OSPFv2 carries the IPv4 VTEP loopback routes (`10.255.0.x/32`) that VXLAN
   encapsulation depends on. Even with a correct frr.conf, VXLAN stayed dark
   until ospfd ran.

Failure chain: no underlay → loopbacks unreachable → corosync (over
`fd00:0:0:ffff::N`) no quorum → Ceph daemons (public net `fc00:20::/64`, also
fabric-routed) failed into systemd start-limit backoff → CephFS mounts failed
→ HA could not start VMs → cluster-101 down.

## Recovery (in order)

1. Restored each node's `/etc/frr/frr.conf` from its own `frr.conf.sav`
   (FRR's write-memory backup, Feb 2026 — newest known-good) + `systemctl
   restart frr` → OSPF6/iBGP/EDGE re-established, quorum restored.
2. `sed ospfd=no→yes` in `/etc/frr/daemons` + restart frr → OSPFv2 up →
   v4 VTEP routes → VXLAN forwarding (verified with tcpdump: etcd peer
   traffic across vni 10101).
3. `systemctl reset-failed` + start for all `ceph-mgr@`/`ceph-mds@`/
   `ceph-osd@` units (start-limit backoff) → HEALTH_OK.
4. Remounted `/mnt/pve/resources` (CephFS) on all nodes.
5. HA services were in `error` state (retries exhausted):
   `ha-manager set vm:X --state disabled` then `--state started` → all six
   cluster-101 VMs booted; etcd 3/3, all nodes Ready.

## Durable fixes (in place, verified)

- Each node's proven `frr.conf.sav` installed as `/etc/frr/frr.conf.local`
  (stale ones backed up as `frr.conf.local.stale-20251210.bak`).
- systemd drop-in `/etc/systemd/system/frr.service.d/10-enable-ospfd.conf`
  re-enables `ospfd=yes` via ExecStartPre on every FRR start.
- **Verified** by triggering the exact failure path: `pvesh set /cluster/sdn`
  (cluster-wide regeneration) — all three nodes regenerated to the full
  config (underlay present, ospfd running) with quorum/Ceph/EVPN/k8s healthy
  throughout.
- Remaining proof point: an actual node reboot exercises the same code path
  and is expected to pass; do one at leisure.

## Open items

- The FRR underlay config lives only on the nodes (+ their .sav/.local
  copies). It should be captured in the repo (`FRR/` symlinks a separate
  repo) or generated by Terraform so a node rebuild can restore it.
- **EVPN hairpin bug** (documented in site.yaml: vnet LXC ↔ services on its
  own host) — now substantially diagnosed via kfree_skb drop-reason tracing:
  - Forward path: the vrf routes host-bound traffic out `vmbr0.10`; the
    bridge delivers it locally AND the frame egresses the physical port and
    gets reflected by the switch → **client segments arrive duplicated**.
  - The duplicates land on the server socket out-of-state →
    `TCP_INVALID_SYN` / `TCP_OLD_SEQUENCE` drops → challenge-ACK/`TCP_RESET`
    storm kills the connection right after the handshake. ("TCP connects,
    payload never flows" fully explained.)
  - Partial fix proven: /32 routes through the `xvrf_evpnz1`/`xvrfp_evpnz1`
    pair make the path symmetric and local at the packet level; one residual
    anomaly (client ACKs still taking the old path for reasons not yet
    pinned) needs a final session before making those routes permanent
    (candidate home: the ansible frr role / ifupdown post-up).
  - Workaround remains: management API access targets pve02
    (`site.yaml` `proxmox.api_endpoint`).
- **New finding (separate)**: an IPv6 forwarding loop for cluster-101 pod
  prefixes (`fd00:101:224::/60`) between the default vrf and the edge -
  steady `ip6_forward` IP_INHDR/hop-limit-exceeded drops and ICMP6
  time-exceeded loops on pve01. Harmless-looking but constant; worth a look
  at the RM_EDGE export/import maps for the pod aggregate.
- Consider a UPS or staggered-boot policy: all three nodes rebooting
  simultaneously is the worst case for a self-hosted routing fabric.
